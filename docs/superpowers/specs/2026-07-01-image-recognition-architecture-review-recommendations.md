## 1. 核心校准:重新理解三个约束

### 1.1 约束对设计的影响

| 约束 | 推论 | 对原规范的影响 |
|---|---|---|
| **仅公司内部使用** | §13 无鉴权完全合理,不需要补 OAuth/JWT/API Key | 原规范安全章节方向正确,但实现细节仍可简化 |
| **高可落地** | 选型倾向"少组件、开箱即用、文档齐全" | 警惕微服务化、中台化、K8s 原生过度设计 |
| **避免臃肿** | 二阶段特性(M2 之后的)是该砍就砍 | 40+ 算法的注册表抽象合理,但配套管理 UI 是臃肿源头 |
| **后续并入大系统** | 横切关注点(鉴权、网关、多租户)由大系统统一处理 | 本系统只做核心业务能力,集成边界要清晰 |

### 1.2 关键校准结论

- 之前对"安全设计单薄"的批评在内部场景下**不成立**;真正要补的是**审计可追溯 + 异常可发现**(事后追责 > 事前拦截)
- 之前对"无 SLO"的批评**仍成立**,但目标应是"工作时段 > 99%",不是云级 99.9%
- 之前对"前端臃肿"的警告**升级为最优先关注点**——Vue + ECharts 是 M0 阶段最大的拖累

> V1.0 遵循"先跑通、再完善"原则,**主动放弃**以下特性以控制范围、保证 M0-M1 阶段 2 周可上线:

### 2.2 新增 §1.7"V1.0 验收标准(SLO)"

> ### 1.7 V1.0 验收标准(新增)
>
> | 指标 | 目标 | 测量方式 |
> |---|---|---|
> | 单图识别 P95 延迟(含队列) | < 30s | Prometheus histogram |

### 2.3 在 §6 数据模型附近新增状态机图

在 §6.1 `inspections` 表描述后,新增如下小节:

> #### 6.1.1 状态机(新增)
>
> ```
> PENDING → RUNNING → SUCCESS
>                  → FAILED (可重试,自动重试 ≤ 3 次)
>                  → DEAD   (超过 max_retries,需手动干预)
>
> 主任务 SUCCESS 后,LLM 富化独立流转:
> ENRICHING → ENRICHED / ENRICH_FAILED
> ```
>
> **关键约束**:
> - 重试期间状态 = `RUNNING`(前端展示"识别中",不引入独立的 `RETRYING` 状态)
> - `status`(主任务)与 `enrichment_status`(LLM 富化)是两个**独立维度**:
>   - 富化失败时,主 `status` 仍为 `SUCCESS`,`enrichment_status` 为 `ENRICH_FAILED`
>   - App 端展示"识别完成,智能建议生成中",并提供 `/enrich` 端点触发重试
> - `DEAD` 状态记录 `error_message` 供排查,前端显示"重试"按钮调用 `/records/{id}/retry`

## 3. 评审会上重点讨论(需要决策)

### 3.1 看板前端方案(关键决策)

| 方案 | 优势 | 劣势 | 建议 |
|---|---|---|---|
| **A. FastAPI 模板 + HTMX** | 零构建、与后端同语言、增量交互好 | 复杂图表难做 | **推荐(M0-M1)** |
| B. Streamlit | 纯 Python、数据展示快 | 定制 UI 难、性能一般 | 备选 |
| C. Vue 3 + ECharts | 体验好、生态全 | 构建链复杂、招人贵 | **不推荐 M0-M1** |

**决策依据**:M0 阶段目标是 1 周跑通端到端,前端是最大的拖累;内部工具核心是"功能可用",不是"体验完美"。

### 3.2 未来并入大系统的边界

**讨论议题**:哪些能力由大系统统一提供,本系统不需要做?

| 能力 | 本系统是否要做 | 由谁提供 |
|---|---|---|
| 用户鉴权(SSO/OAuth) | **不做** | 大系统 |
| API 网关(限速/熔断) | **不做** | 大系统 |
| 多租户隔离 | **不做** | 大系统 |
| 审计日志收集 | 本系统写本地文件 | 大系统集中采集 |
| 监控告警通道(钉钉/邮件) | 本系统只暴露指标 | 大系统统一告警 |
| 文件存储(MinIO) | 本系统自管 | 并入时迁移到统一 OSS |
| 业务核心能力(识别/富化/报告) | **本系统做** | — |

**建议在 §13 末尾新增**:

### 4.3 代码层关键简化点

#### 4.3.1 Celery 临时文件管理

主规范未提及 Celery worker 从 MinIO 下载文件后的临时文件处理,建议在 §5.4 任务队列章节补充:

> **临时文件策略**:
> - worker 处理前下载到 `tempfile.NamedTemporaryFile` 创建的本地文件
> - 任务完成后立即删除(try/finally)
> - 失败重试时重新下载(简单可靠)
> - **不引复杂的临时文件管理库**
> - Celery `result_backend` **可省**,只保留 `broker_url`(主规范 §10.1 用了 result_backend,实际不需要)

#### 4.3.2 docker-compose 服务精简

**主规范 §16.1 现状**:9 个服务

**精简后 6 个服务**:

| 服务 | 说明 |
|---|---|
| `postgres` | 不变 |
| `redis` | 不变 |
| `minio` | 不变 |
| `minio-init` | 不变 |
| `backend` | FastAPI(8000) |
| `worker` | Celery worker(单实例多队列:`-Q inspect_queue,stats_queue,cleanup_queue`) |
| `frontend` | 极简内部工具页(端口 8000 下的子路径,与 backend 同容器或独立) |

#### 4.3.3 算法注册表刷新机制

**主规范 §5.2 现状**:"updated_at 变更触发热刷新"(机制未指定)

**简化为**:
- 启动时加载到内存字典(`{code: AlgorithmRow}`)
- 修改算法 = 滚动重启 worker(K8s 场景下成本可接受,本地直接重启)
- 文档注明"算法变更后需重启 worker"
- **不实现**:LISTEN/NOTIFY、Redis pub/sub、文件 watch

**理由**:内部场景下改算法频率极低(月度级别),省 2-3 天开发量。

## 6. 关键架构调整总结

### 6.1 看板技术栈调整

| 阶段 | 技术栈 |
|---|---|
| M0-M1 | FastAPI 模板 + HTMX(增量交互)或 Jinja2 + Alpine.js |
| M2 海康超脑 PoC | 同上 |
| M3+ 规模化 | 此时再评估 Vue / 保留极简方案 |

**理由**:3 人内部系统,前端是最大拖累;Vue 项目的 npm 依赖、构建配置、组件库升级消耗精力远超价值。

### 6.2 安全设计简化

**内部场景下的真正安全需求**:

| 需求 | 实现方式 | 不做的事 |
|---|---|---|
| 文件类型校验 | MIME + 文件头 | 复杂文件杀毒扫描 |
| 凭据管理 | `.env` 文件(本地) | K8s Secret / KMS |
| 数据脱敏 | LLM 输入字段白名单(代码层) | 人脸检测、自动化脱敏工具 |
| 审计可追溯 | `audit_logs` 表记录关键操作 | ELK 集中日志 |
| 异常可发现 | Prometheus 指标 + 健康检查 | 复杂告警规则 |

**关键判断**:
- "X-Inspector-Id 可伪造"在内部场景下**不是问题**——内部员工有合规约束,出问题能追责
- "网络安全靠内网 + VPN"**完全够**——并入大系统时由大系统网关统一处理
- 纵深防御(mTLS、设备指纹、请求签名)在 V1.0 阶段**完全不需要**

### 6.3 部署简化

**V1.0 阶段目标**:单机能跑、资源占用低、维护简单。

| 资源 | 建议配置 |
|---|---|
| 本地开发机 | 8GB RAM 即可启动全部服务 |
| Celery worker 并发 | inspect_queue 4-8,stats_queue 1-2(单 worker 多队列) |
| 监控 | 只暴露 Prometheus 指标,不部署 Grafana(本地) |

**云部署延后**:M3 之后再说,本地跑通是 V1.0 的成功标准。

## 7. 里程碑工作量重新评估

### 7.1 原 M1 估算的问题

**原 M1**:"1-2 周增加 1-2 个算法"

**问题**:每个新算法涉及 prompt 调优、样本收集、引擎配置,**通常 1 周仅够上线 1 个生产可用算法**。

**重新评估**:
- M0(1 周):脚手架 + 1 个算法端到端
- M1(1 周):增加 1-2 个算法(基于 M0 的复用模板,工作量可控)

**总周期**:2 周可上线 V1.0。

### 7.2 风险点与应对

| 风险 | 应对 |
|---|---|
| M0 第一周完不成 | 进一步砍特性:看板只做"列表 + 详情"两个页面,重试按钮都用 SQL 触发 |
| LLM 富化效果差 | 准备 10 条样本手工调 Prompt,V1.0 上线后看真实数据再迭代 |
| 引擎适配器集成踩坑 | 选择 1 个有 Python SDK 的云厂商(如通义千问),降低集成成本 |
| 海康超脑文件级分析不可行 | M2 PoC 失败则保持 M1 状态,V2 再说 |
