## 1. 必须决策 (Q1-Q4)

### Q1: 看板前端技术栈

**问题**: 主规范 §3.1 推荐 FastAPI 模板 + HTMX/Alpine.js, 实际选择 Vue 3 + Element Plus。

**决策**: **采用 Vue 3 + Element Plus** (偏离主规范推荐)

**理由**:
- 团队熟悉 Vue, 招人容易
- Element Plus 组件库开箱即用 (表单/对话框/抽屉/表格)
- 复杂算法配置表单需要响应式状态管理
- 主规范 §3.1 的"备选 A"考虑了 Streamlit, Streamlit 性能/定制差

**接受的代价**:
- 前端构建链复杂 (npm, Vite, 依赖管理)
- 主规范 §6.1 警告的"前端臃肿" 风险

**M3+ 缓解**: 如果 40+ 算法或大屏需求出现, 重构为 Vue 3 + ECharts (主规范 ADR-007)

**决策状态**: ✅ 已实现 (V1.0)

## 2. 建议决策 (Q5-Q8)

### Q5: LLM 富化触发位置

**问题**: 富化在 Celery 任务内同步调用, 还是独立队列?

**决策**: **Celery 任务内同步调用** (主规范 ADR-005)

**理由**: 失败隔离, 流程简单, 后续可拆

**实施** (`backend/app/tasks/inspection.py`):
- 同步模式 (TASK_SYNC_MODE=true): 主任务直接 await 富化
- 异步模式 (TASK_SYNC_MODE=false): Celery send_task 到 `stats_queue`

**决策状态**: ✅ 已实现 (ADR-005)

---

### Q6: 算法注册表刷新机制

**问题**: 启动加载 + 滚动重启, 还是热刷新?

**决策**: **启动加载 + 滚动重启** (主规范 §4.3.3 + ADR-006)

**理由**: 内部场景改算法频率低, 省 2-3 天开发量

**实施**:
```python
# backend/app/main.py lifespan()
async with sm() as session:
    registry = get_registry()
    await registry.load(session)
```

**不实现**: LISTEN/NOTIFY, Redis pub/sub, 文件 watch

**决策状态**: ✅ 已实现 (ADR-006)

---

### Q7: LLM 厂商选型

**问题**: 主规范 §3.1 推荐通义千问 qwen-plus, 实际选择?

**决策**: **不绑死厂商**, 默认 MiniMax M3, 可通过 env 切换 (ADR-004)

**理由**:
- 内部按成本/合规切换, 不锁死
- OpenAI 兼容接口是标准

**当前默认**: MiniMax M3 (推理能力强)

**切换示例**:
```bash
LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL=qwen-vl-plus
```

**决策状态**: ✅ 已实现, 默认 MiniMax (偏离主规范 §3.1 推荐, ADR-004 说明)

## 3. 后续讨论 (Q9-Q10)

### Q9: 未来并入大系统的边界

**问题**: 主规范 §3.2 列出的能力划分, 是否落地?

**决策**: **已落地** (`docs/superpowers/integration-boundary.md`)

**核心结论**:

**决策状态**: ✅ 已文档化

### Q10: V2.0 鉴权策略

**问题**: V2.0 鉴权由本系统还是大系统统一?

**决策**: **大系统统一** (主规范 §1.6 + §3.2)

**理由**: 横切关注点由大平台处理, 本系统只做业务能力

**实施**:
- V1.0: 仅 `X-Inspector-Id` 业务标识校验
- V2.0: 大平台网关注入 `X-Inspector-Id`, 本系统不感知
- 本系统代码: 保留 `X-Inspector-Id` 校验 (防御性)

**决策状态**: ✅ 已规划, V1.0 已实现基础

## M2 关键决策 (2026-07-07)

### D-014: 启用 TUS 断点续传

- **背景**: cpolar 公网映射后大文件上传失败率 ~100%
- **决策**: 自研 TUS 1.0.0 客户端 + 服务端, 0 npm 依赖, ~500 行
- **理由**: 现状 (cpolar 1-3Mbps) 用 tus-js-client over-engineered, 5MB/分片 + 5 次重试 + 24h GC 足够
- **度量**: 5MB 照片失败率 100% → 0%, 100MB 视频 不可用 → 15 分钟可传完
- **ADR**: [ADR-014](./adr.md)

### D-015: 客户端 Canvas 压缩图片

- **背景**: 手机相册原图 5-15MB 在 cpolar 弱网必超时
- **决策**: 选完文件后浏览器 Canvas 缩放 + JPEG 0.85
- **理由**: 5MB → 500KB (10x), 视觉无感, 解决 80% 用户场景
- **例外**: HEIC (iPhone 原生) 在 Chrome/Firefox 不能 canvas 解码, 透传给后端
- **度量**: 5MB iPhone 照片 40s (失败) → 1.5s (成功)
- **ADR**: [ADR-015](./adr.md)

### D-016: 修文件大小按 file_type 分支 (M1 bug)

- **背景**: M1 视频被 20MB 拒, config 早已定义 max_video_size=500MB 但代码不用
- **决策**: inspect.py 按 ext 分支选 max_size
- **理由**: 改 5 行, 立即生效
- **教训**: 加新配置时, 必须搜所有用到的代码, 不能只加不用
- **ADR**: [ADR-016](./adr.md)

### D-017: 部署走 cpolar (公网映射), 不自建服务器

- **背景**: 用户想用真实手机访问, 验证公网可行性
- **决策**: 用 cpolar 内网穿透映射 frontend + backend
- **理由**:
  - 零运维 (cpolar 免费版足够 demo)
  - 真实测试 cpolar 弱网下表现
  - 未来切正式服务器时, 只需换域名, 业务代码无改
- **风险**: cpolar 免费版带宽 1-3Mbps, 公网延迟 ~50ms
- **应对**: 正是 TUS + 压缩要解决的问题
